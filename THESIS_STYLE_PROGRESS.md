# 校园跑腿系统（类论文）进度跟踪文档

> 用途：以“论文写作 + 工程落地”双线方式推进项目，保证每个阶段有产出、可验收、可复盘。

---

## 1. 项目基本信息

- 项目名称：校园跑腿系统
- 项目定位：校园内即时跑腿服务平台（用户下单、骑手接单、订单履约、评价与管理）
- 技术栈：
  - 前端：Vue 3
  - 后端：Spring Boot / Spring Cloud（分阶段演进）
  - 数据库：PostgreSQL
  - 缓存：Redis
  - 部署：Docker Compose
- 当前策略：先“可拆分单体”跑通 MVP，再按边界拆分微服务

---

## 2. 研究目标与工程目标

### 2.1 研究目标（论文视角）

- 明确校园跑腿业务流程与角色协作模型
- 设计可扩展的系统架构与核心数据模型
- 验证系统在核心业务链路上的可行性与稳定性

### 2.2 工程目标（项目视角）

- 在最短周期内跑通 MVP 闭环：下单 → 接单 → 执行 → 完成 → 评价
- 建立可迭代的代码结构、接口规范与交付流程
- 保证每次提交聚焦单一职责，降低变更风险

## 6. 提交纪律（单一职责）

### 提交原则

- 一次提交只做一件事：一个小功能 / 一个修复 / 一份文档
- 提交信息必须可读：`type(scope): message`
- 每次提交都要能独立回滚

### 建议的提交类型

- `feat`: 新功能
- `fix`: 缺陷修复
- `docs`: 文档更新
- `refactor`: 重构（无功能变化）
- `test`: 测试相关
- `chore`: 工程维护

### 示例

- `docs(plan): add thesis-style progress tracker`
- `feat(order): create order api endpoint`
- `fix(order): block invalid status transition`

---

## 7. 风险清单与应对

- 风险：过早微服务化导致开发效率下降
  - 应对：先模块化单体，满足条件再拆分
- 风险：需求边界不清导致反复返工
  - 应对：先冻结 MVP 范围，新增需求进入 backlog
- 风险：提交粒度过大导致回滚困难
  - 应对：强制单一职责提交
