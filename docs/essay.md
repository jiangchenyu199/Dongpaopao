# dongpaopao

## 1. 项目章程

### 1.1 项目目标
为校园场景提供安全、可追踪、可审核的跑腿服务平台。

### 1.2 MVP 必做功能
- 用户发单
- 跑腿员自主接单（非派单）
- 订单状态流转（待接单、进行中、已完成、已取消、有争议）
- 管理员 Web 端审核与处理
- 评价与基础投诉入口

### 1.3 MVP 暂不做
- 在线支付/结算
- 实时地图轨迹
- 复杂消息推送系统
- 微服务拆分（先保持单体架构）

### 1.4 角色定义
- 普通用户：可发单
- 跑腿员：可接单
- 管理员：Web 端管理与审核

说明：同一小程序账号可同时具备普通用户和跑腿员能力，前端通过模式切换降低使用复杂度。

### 1.5 成功标准
- 主流程闭环可跑通：发单 -> 接单 -> 完成 -> 评价
- 管理员可在 Web 端查看并处理订单/用户
- OpenAPI 文档可访问（OpenAPI 3.1 + springdoc）
- PostgreSQL / Redis / JWT 链路可运行

## 2. 功能清单（MoSCoW）

### 2.1 Must（MVP 必须有）
1. 账号登录与 JWT 鉴权（小程序端、管理端）。
2. 同一账号可双能力：可发单、可接单（支持前端模式切换）。
3. 发单：填写取件地、送达地、描述、赏金、截止时间。
4. 自主接单：跑腿员从待接单池抢单/领单（非派单）。
5. 订单状态流转：`待接单 -> 进行中 -> 已完成`，并支持取消与争议分支。
6. 基础评价：发单方对完成订单评分与文字评价。
7. 管理员 Web：用户管理、订单管理、投诉处理入口。
8. OpenAPI 接口文档可访问（springdoc）。

### 2.2 Should（建议做）
1. 接单资格开通机制（例如学生认证标记）。
2. 接单限制策略（进行中订单上限、禁止接自己单）。
3. 订单筛选与搜索（按状态、时间、关键字）。
4. 管理后台统计看板（日订单量、完成率、取消率）。
5. 关键行为操作日志（审计）。

### 2.3 Could（可选增强）
1. 站内消息通知中心。
2. 标签化任务类型（代取快递、代买、代送文件）。
3. 信誉分机制（完成率、投诉率）。
4. 管理端导出报表。

### 2.4 Won't（本阶段不做）
1. 在线支付与自动分账。
2. 实时地图轨迹。
3. 微服务拆分与复杂治理（网关、注册中心、配置中心等）。
