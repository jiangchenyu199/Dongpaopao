# dongpaopao

## 1. 项目章程

### 1.1 项目目标
为校园场景提供安全、可追踪、可审核的跑腿服务平台。

### 1.2 MVP 必做功能
- 用户发单
- 跑腿员自主接单（非派单）
- 订单状态流转（待接单、进行中、已完成、已取消、有争议）
- 管理员 Web 端审核与处理
- 评价与基础投诉入口

### 1.3 MVP 暂不做
- 在线支付/结算
- 实时地图轨迹
- 复杂消息推送系统
- 微服务拆分（先保持单体架构）

### 1.4 角色定义
- 普通用户：可发单
- 跑腿员：可接单
- 管理员：Web 端管理与审核

说明：同一小程序账号可同时具备普通用户和跑腿员能力，前端通过模式切换降低使用复杂度。

### 1.5 成功标准
- 主流程闭环可跑通：发单 -> 接单 -> 完成 -> 评价
- 管理员可在 Web 端查看并处理订单/用户
- OpenAPI 文档可访问（OpenAPI 3.1 + springdoc）
- PostgreSQL / Redis / JWT 链路可运行

## 2. 功能清单（MoSCoW）

### 2.1 Must（MVP 必须有）
1. 账号登录与 JWT 鉴权（小程序端、管理端）。
2. 同一账号可双能力：可发单、可接单（支持前端模式切换）。
3. 发单：填写取件地、送达地、描述、赏金、截止时间。
4. 自主接单：跑腿员从待接单池抢单/领单（非派单）。
5. 订单状态流转：`待接单 -> 进行中 -> 已完成`，并支持取消与争议分支。
6. 基础评价：发单方对完成订单评分与文字评价。
7. 管理员 Web：用户管理、订单管理、投诉处理入口。
8. OpenAPI 接口文档可访问（springdoc）。

### 2.2 Should（建议做）
1. 接单资格开通机制（例如学生认证标记）。
2. 接单限制策略（进行中订单上限、禁止接自己单）。
3. 订单筛选与搜索（按状态、时间、关键字）。
4. 管理后台统计看板（日订单量、完成率、取消率）。
5. 关键行为操作日志（审计）。

### 2.3 Could（可选增强）
1. 站内消息通知中心。
2. 标签化任务类型（代取快递、代买、代送文件）。
3. 信誉分机制（完成率、投诉率）。
4. 管理端导出报表。

### 2.4 Won't（本阶段不做）
1. 在线支付与自动分账。
2. 实时地图轨迹。
3. 微服务拆分与复杂治理（网关、注册中心、配置中心等）。

## 3. 系统设计

### 3.1 设计目标
本章围绕校园跑腿场景的 MVP 实现，建立“可快速落地、可稳定运行、可平滑扩展”的系统设计方案。系统设计目标如下：

1. 支撑核心业务闭环：发单、接单、履约、评价、争议处理。
2. 保证权限与数据安全：统一鉴权、角色分权、关键操作留痕。
3. 控制实现复杂度：采用单体架构分层实现，为后续微服务拆分预留边界。
4. 提升后续演进能力：模块职责清晰，接口契约可复用。

### 3.2 总体架构设计
系统采用前后端分离架构，由小程序端、管理员 Web 端与后端服务组成，基础设施依赖 PostgreSQL 与 Redis。

- 小程序端（uniapp）：面向学生用户，提供发单、接单、订单跟踪、评价等功能。
- 管理员 Web 端（Vue3）：面向管理员，提供用户管理、订单管理、争议处理与运营看板功能。
- 后端服务（Spring Boot 4）：提供统一 REST API、鉴权、业务编排与数据访问能力。
- PostgreSQL：存储用户、订单、评价、争议、审计日志等结构化数据。
- Redis：缓存热点数据、登录态辅助信息与幂等控制信息。

在该架构下，前端通过 OpenAPI 契约与后端交互，后端对外提供统一 API 入口，对内按业务域分层组织代码。

### 3.3 前端设计

#### 3.3.1 小程序端设计
小程序端采用“单账号双能力”模型，同一账号可在下单与接单模式间切换。模式切换仅影响前端导航与默认页面聚焦，不改变账号身份本身。

核心页面建议如下：
- 首页：根据当前模式展示推荐入口与待办事项。
- 发单页：创建跑腿订单。
- 接单大厅页：浏览待接单池并执行接单。
- 订单页：区分“我发布的订单”和“我接的订单”。
- 我的页：账号资料、接单资格、历史评价。

#### 3.3.2 管理员 Web 端设计
管理员 Web 端采用后台管理信息架构，主要模块如下：
- 用户管理：查询、禁用、恢复、风险标记。
- 订单管理：状态筛选、详情查看、人工干预。
- 争议处理：查看证据、裁定结果、状态回写。
- 运营看板：订单量、完成率、取消率、争议率等基础指标。

### 3.4 后端分层与模块设计
后端采用典型分层设计，划分为接口层、应用服务层、领域规则层与数据访问层。

- 接口层（Controller）：处理请求参数、鉴权上下文、响应封装。
- 应用服务层（Service）：组织业务流程与事务边界。
- 领域规则层（Domain/Policy）：封装状态流转约束、接单规则、争议裁定规则。
- 数据访问层（Repository）：负责 PostgreSQL 持久化操作。

结合 MVP 需求，建议划分以下业务模块：
1. 认证与账号模块：登录、JWT、角色与权限。
2. 订单模块：发单、接单、状态流转、取消、完成。
3. 评价与争议模块：评价提交、投诉/争议创建、处理结果落库。
4. 管理模块：后台审核、封禁策略、审计日志查询。

### 3.5 数据设计（概念层）
MVP 阶段建议先构建最小可用数据模型，核心实体包括：

- 用户（User）：账户基础信息、状态、接单资格标识。
- 订单（Order）：发布者、接单者、地址信息、赏金、状态、时间信息。
- 评价（Review）：订单关联、评价人、被评价人、评分与评价内容。
- 争议（Dispute）：订单关联、发起人、争议原因、处理状态、裁定结果。
- 审计日志（AuditLog）：关键操作留痕（操作者、动作、时间、对象）。

其中订单状态集合采用：`PENDING`、`IN_PROGRESS`、`COMPLETED`、`CANCELLED`、`DISPUTED（有争议）`。

### 3.6 安全与权限设计

1. 认证机制：基于 JWT 的无状态认证，区分小程序端与管理端登录上下文。
2. 鉴权机制：接口按角色与权限点进行访问控制，管理端采用更细粒度权限策略。
3. 业务风控规则：禁止接自己发布的订单，限制进行中订单数量，防止恶意刷单。
4. 审计机制：争议处理、封禁、订单强制状态变更等高风险操作必须记录审计日志。

### 3.7 部署与运行设计（开发阶段）
开发环境采用本地或容器化部署方式：

- 应用服务：Spring Boot 4 单体服务。
- 数据服务：PostgreSQL。
- 缓存服务：Redis。
- 管理员前端：Vue3 独立开发服务器。

通过环境变量区分开发、测试与生产配置，降低环境耦合。API 文档由 springdoc 自动生成，作为联调与验收依据。

### 3.8 架构演进预留
在不增加 MVP 实现复杂度的前提下，系统预留以下演进路径：

1. 先在单体内完成按业务域拆包，形成稳定边界。
2. 后续优先拆分订单与用户模块，逐步演进至微服务。
3. 引入网关、配置中心、服务注册与链路追踪等治理能力。
4. 在支付、通知等高耦合场景中引入事件驱动机制。

## 4. 系统实现

### 4.1 实现环境与技术栈落地
本系统在 MVP 阶段采用“前后端分离 + 单体后端”实现方式，具体技术栈落地如下：

- 小程序端：uniapp（Vue3 语法）
- 管理员 Web 端：Vue3 + 状态管理（如 Pinia）+ 路由管理
- 后端：Spring Boot 4（REST API）
- 数据库：PostgreSQL
- 缓存：Redis
- 认证：JWT
- 接口文档：OpenAPI 3.1 + springdoc

该实现方案兼顾了开发效率与后续可扩展性，能够在较低复杂度下验证核心业务闭环。

### 4.2 后端关键模块实现

#### 4.2.1 认证与权限实现
后端通过统一认证过滤链实现 JWT 校验。用户登录成功后签发访问令牌，后续请求通过令牌携带身份信息进入业务层。系统将用户身份与权限点进行绑定，区分普通用户、跑腿员与管理员接口访问范围。

在实现上，建议将权限控制分为两层：
1. 接口层权限：限制不同角色访问不同 API。
2. 业务层权限：对关键动作进行二次校验（例如接单资格、封禁状态校验）。

#### 4.2.2 订单主流程实现
订单主流程围绕“发单、接单、履约、完成、评价”组织，核心实现点如下：

1. 发单：校验入参合法性，创建订单并写入 `PENDING` 状态。
2. 接单：在事务中校验状态与接单人资格，更新接单人并变更为 `IN_PROGRESS`。
3. 完成：校验订单当前状态与操作者身份，写入完成时间并变更为 `COMPLETED`。
4. 评价：仅允许对已完成订单评价，防止重复评价。

为避免并发接单导致的重复领取，应采用“事务 + 条件更新 + 唯一约束”组合策略保证一致性。

#### 4.2.3 争议处理实现
当履约过程中发生争议时，订单由 `IN_PROGRESS` 进入 `DISPUTED`，并生成争议记录。管理员在后台完成裁定后，将订单最终状态更新为 `COMPLETED` 或 `CANCELLED`。争议处理过程中的关键动作均写入审计日志，保证可追溯性。

#### 4.2.4 管理后台实现
管理员 Web 端对应后端管理 API，主要实现：
- 用户状态管理（启用、禁用、限制接单）
- 订单查询与人工干预
- 争议工单处理与裁定结果回写
- 基础运营数据查询

管理接口应独立权限分组，避免普通用户令牌越权访问。

### 4.3 前端关键功能实现

#### 4.3.1 小程序端实现
小程序端采用“模式切换 + 功能分区”实现：

- 下单模式：聚焦发布订单、查看已发布订单、评价入口。
- 接单模式：聚焦待接单大厅、进行中订单、履约确认。

模式切换仅改变界面导航与默认查询维度，不改变账号实体。订单页通过“我发布的/我接的”双列表满足同账号多角色并行使用场景。

#### 4.3.2 管理员 Web 端实现
管理员端以列表页 + 详情页 + 操作弹窗为核心交互结构，实现用户、订单、争议三类管理能力。针对高风险操作（封禁、争议裁定、强制改状态）需提供二次确认与操作备注输入，便于审计。

### 4.4 数据库实现要点
MVP 阶段重点保障订单读写正确性与可追踪性，数据库实现建议如下：

1. 订单表保留完整状态字段、时间字段与操作者字段。
2. 争议表与订单表建立稳定关联，支持追溯争议发起与处理结果。
3. 评价表限制同一订单评价唯一性，防止重复提交。
4. 审计日志表记录关键操作快照，支持后台追责与问题定位。
5. 对高频查询条件建立索引（状态、发布时间、接单人、发布人）。

### 4.5 接口文档与联调实现
通过 springdoc 自动生成接口文档，统一暴露 `/v3/api-docs` 与 Swagger UI 页面。前后端以 OpenAPI 契约作为联调基准，减少因字段命名、状态枚举不一致导致的沟通成本。

在实现阶段，建议建立“接口变更即更新文档”的约束，确保文档与代码同步。

## 5. 系统测试与结果分析

### 5.1 测试目标与范围
本章测试目标是验证系统是否满足 MVP 阶段的核心业务与质量要求。测试范围覆盖：

1. 功能正确性：主流程与异常流程。
2. 权限安全性：鉴权、越权、封禁状态校验。
3. 状态一致性：订单状态流转合法性与并发接单一致性。
4. 可用性：管理员后台核心操作可用。

### 5.2 功能测试

#### 5.2.1 主流程测试
对“发单 -> 接单 -> 完成 -> 评价”进行端到端验证，期望结果为：
- 每一阶段状态变更准确；
- 页面与接口返回一致；
- 关键数据正确落库。

#### 5.2.2 异常流程测试
针对典型异常场景进行验证：
- 非法参数发单（缺失字段、截止时间非法）；
- 重复接单或接已被领取订单；
- 非订单关联用户尝试完成订单；
- 已完成订单重复评价。

期望系统返回明确错误码与可读错误信息，不出现脏数据。

#### 5.2.3 争议流程测试
模拟履约争议触发，验证如下链路：
`IN_PROGRESS -> DISPUTED -> (COMPLETED | CANCELLED)`。
测试重点包括争议记录创建、管理员裁定生效、审计日志完整性。

### 5.3 权限与安全测试
权限测试重点如下：

1. 未登录请求访问受保护接口应被拒绝。
2. 普通用户不得访问管理员 API。
3. 被禁用账号不得执行发单或接单动作。
4. 高风险后台操作必须记录操作人和时间。

测试结果应表明系统具备基础权限隔离能力，能够降低越权访问风险。

### 5.4 一致性与性能初测

#### 5.4.1 一致性测试
通过并发接单测试验证同一订单不会出现多个接单人。若并发场景下仍只存在唯一有效接单记录，说明“事务 + 约束”策略有效。

#### 5.4.2 性能初测
以 MVP 核心接口（发单、接单、订单查询）进行基础压力测试，观察平均响应时间与 P95 指标。测试结果用于确认当前架构可支撑校园初期规模，并为后续优化提供基线。

### 5.5 测试结论与问题分析
综合测试结果，系统可完成既定 MVP 功能目标，主流程与争议流程均可闭环。当前主要优化方向包括：

1. 提升复杂条件查询性能（索引与分页策略优化）。
2. 增强异常提示与用户引导文案。
3. 完善监控告警能力（接口耗时、错误率、争议量）。

## 6. 总结与展望

### 6.1 工作总结
本文围绕校园跑腿场景，从需求分析、系统设计到实现与测试，完成了 dongpaopao 项目的 MVP 方案构建。系统实现了用户发单、跑腿员自主接单、订单状态流转、评价与争议处理、管理员后台管理等核心能力，验证了“单体分层 + 前后端分离”路线在初期阶段的可行性。

同时，本文明确了“同一账号双能力 + 前端模式切换”的角色组织方式，使系统更贴合学生群体“一人多角色并行”的真实使用习惯。

### 6.2 主要贡献
本阶段工作的主要贡献可归纳为：

1. 建立了面向校园场景的 MVP 需求框架与优先级体系。
2. 形成了包含争议处理机制的订单闭环设计。
3. 给出了可演进的系统架构方案，为后续微服务化提供边界基础。
4. 通过测试验证了核心流程与权限控制的可用性。

### 6.3 局限性
受 MVP 范围控制，系统仍存在以下局限：

1. 尚未接入在线支付与资金结算能力。
2. 缺少实时通知与轨迹能力，用户体验仍有提升空间。
3. 监控与治理能力处于基础阶段，尚未形成完整生产级运维体系。
4. 当前为单体部署，尚未验证大规模并发下的服务治理效果。

### 6.4 后续展望
后续可围绕以下方向持续推进：

1. 业务能力扩展：接入支付、消息通知、信誉分体系。
2. 架构能力升级：逐步拆分订单、用户、通知模块，演进至 Spring Cloud 架构。
3. 数据能力建设：建立更完整的数据分析指标体系，支撑运营决策。
4. 平台能力完善：开放用户 Web 端，形成多端统一服务体系。
